# Accumulator Feature for Salsa.jl

## Overview

An accumulator is a side-channel for collecting values (typically diagnostics/errors) during `@derived` function execution, queryable from outside afterward. This is modeled after Rust Salsa's `#[salsa::accumulator]` feature.

## Proposed API

### Declaration — a new macro creating an accumulator type

```julia
@declare_accumulator Diagnostics::String
@declare_accumulator Warnings::Warning   # any type works
```

### Pushing values — inside any `@derived` function

```julia
@derived function type_check(rt, item::Item)::CheckResult
    # ... analysis ...
    if problem_found
        Salsa.push_accumulated!(rt, Diagnostics, "undeclared variable 'x'")
    end
    # ... more analysis ...
    return result
end
```

### Querying accumulated values — from outside, after calling a tracked function

```julia
# Get all Diagnostics accumulated during type_check(rt, my_item)
# and all functions it transitively called:
diags = Salsa.accumulated(rt, type_check, Diagnostics, my_item)
# Returns Vector{String}
```

## Architecture & Implementation

### Step 1: Define accumulator type infrastructure

Create a new file `src/accumulators.jl`. An accumulator is identified by a unique type tag (like `DerivedKey{F}` identifies derived functions):

```julia
abstract type AbstractAccumulator{T} end

# Generated by @declare_accumulator Diagnostics::String
struct Diagnostics <: AbstractAccumulator{String} end

# Extract the element type
accum_eltype(::Type{<:AbstractAccumulator{T}}) where {T} = T
```

### Step 2: Add accumulator storage to `TraceOfDependencyKeys`

Each trace already tracks the dependencies of the current `@derived` function. Add a dict to also hold accumulated values.

**File: `src/trace.jl`**

Add a new field to `TraceOfDependencyKeys`:

```julia
mutable struct TraceOfDependencyKeys
    ordered_deps::Vector{DependencyKey}
    seen_deps::Set{DependencyKey}
    lock::Base.ReentrantLock
    call_stack::Union{Nothing,SalsaStackFrame}
    should_trace::Bool

    # NEW: accumulated values pushed during this function's execution
    accumulated::Dict{Type, Vector}   # Type => Vector{T} for each accumulator type
end
```

**Why here and not in `DefaultStorage`?** Because:
- A trace is **per-invocation** of a `@derived` function — exactly the right granularity.
- It's already locked for thread safety (parallel child tasks push deps to the same trace).
- It's already pooled and recycled — accumulator vectors get `empty!`'d on reuse.
- It doesn't affect `_TracingRuntime`'s isbits property (trace is already heap-allocated, accessed by index).

**Constructor change**: Initialize the `accumulated` field in the `TraceOfDependencyKeys()` constructor:

```julia
function TraceOfDependencyKeys()
    N = 30
    return new(
        sizehint!(Vector{DependencyKey}(), N),
        sizehint!(Set{DependencyKey}(), N),
        Base.ReentrantLock(),
        nothing,
        true,
        Dict{Type, Vector}(),   # NEW
    )
end
```

**Pool recycling**: When traces are returned to the pool (in `release_trace_id` / wherever the trace is cleared for reuse), also clear the accumulated dict:

```julia
empty!(trace.accumulated)
```

Add a `collect_accumulated` function (analogous to `collect_trace`):

```julia
function collect_accumulated(trace::TraceOfDependencyKeys)
    # No lock needed — user function is guaranteed to have finished
    return copy(trace.accumulated)
end
```

### Step 3: Implement `push_accumulated!`

**File: `src/accumulators.jl`**

```julia
function push_accumulated!(rt::_TracingRuntime, ::Type{A}, value) where {A <: AbstractAccumulator}
    tr = trace(rt)
    T = accum_eltype(A)
    @lock tr.lock begin
        vec = get!(tr.accumulated, A) do
            Vector{T}()
        end
        push!(vec, convert(T, value))
    end
end

# Error if called from top-level (not inside a @derived function)
function push_accumulated!(rt::_TopLevelRuntime, ::Type{A}, value) where {A}
    error("push_accumulated! can only be called inside a @derived function")
end
```

### Step 4: Store accumulated values alongside `DerivedValue`

Extend `DerivedValue` in `src/default_storage.jl` to store the accumulated values collected during execution:

```julia
mutable struct DerivedValue{T}
    value::T
    dependencies::Vector{DependencyKey}
    changed_at::Revision
    verified_at::Revision

    # NEW: accumulated values from this function's execution (not transitive)
    accumulated::Dict{Type, Vector}
end
```

At the end of `_memoized_lookup_internal`, after `collect_trace(rt)` is called, also collect accumulated values from the trace and store them in the `DerivedValue`:

```julia
deps = collect_trace(rt)
accums = collect_accumulated(rt)  # NEW: extract from trace before it's recycled

# Store in DerivedValue
DerivedValue{RT}(v, deps, current_revision, current_revision, accums)
```

When the function is **re-executed** (invalidated), the accumulated values are replaced with the new execution's values. When the function is **not re-executed** (still valid), the cached accumulated values remain.

### Step 5: Implement `accumulated()` — the transitive query

This is the most interesting part. `accumulated(rt, func, AccumType, args...)` must return **all values** pushed by `func(rt, args...)` **and every function it transitively called**.

**File: `src/accumulators.jl`**

```julia
function accumulated(rt::_TopLevelRuntime, func, ::Type{A}, args...) where {A <: AbstractAccumulator}
    key = DerivedKey{typeof(func)}(args)
    T = accum_eltype(A)
    result = T[]
    _collect_accumulated!(rt, key, A, result, Set{DependencyKey}())
    return result
end

function _collect_accumulated!(rt, key::DependencyKey, ::Type{A}, result, visited) where {A}
    key in visited && return
    push!(visited, key)

    # Ensure value is computed and up-to-date
    memoized_lookup(rt, key)

    storage = Salsa.storage(rt)
    derived_val = _get_cached_value(storage, key)
    derived_val === nothing && return

    # Collect this function's own accumulated values
    if haskey(derived_val.accumulated, A)
        append!(result, derived_val.accumulated[A])
    end

    # Recurse into dependencies (only DerivedKeys, not InputKeys)
    for dep in derived_val.dependencies
        if dep isa DerivedKey
            _collect_accumulated!(rt, dep, A, result, visited)
        end
    end
end
```

**Key property**: Because dependencies are already tracked and stored in `DerivedValue.dependencies`, we can walk the dependency graph without re-executing anything. The `visited` set prevents double-counting in diamond dependency patterns.

**Note**: A `_get_cached_value(storage, key)` helper may need to be added to `DefaultStorage` to look up a `DerivedValue` by key without triggering computation. This is a simple dict lookup into `storage.derived_function_maps`.

### Step 6: Wire into the storage flow

In `src/default_storage.jl`, the `_memoized_lookup_internal` function already has a point where it collects the trace and creates/updates a `DerivedValue`. After the line where `collect_trace(rt)` is called, also collect accumulated values.

Find every location where a `DerivedValue` is constructed or updated and ensure the `accumulated` field is populated:

1. **New computation** (cache miss): Build `DerivedValue` with `accums` from the trace.
2. **Re-computation** (invalidated): Replace `accumulated` on the existing `DerivedValue`.
3. **Still valid** (cache hit): Leave `accumulated` unchanged — it reflects the execution that produced the cached value.

Also ensure that the dependency-swap optimization (where `existing_value.dependencies` is swapped with the trace's `ordered_deps` to reuse the allocation) does NOT swap accumulated — accumulated values should always be freshly collected from the current trace.

### Step 7: The `@declare_accumulator` macro

**File: `src/accumulators.jl`**

Minimal macro — just generates a struct:

```julia
macro declare_accumulator(expr)
    @assert expr isa Expr && expr.head == :(::) "Expected Name::Type, e.g. @declare_accumulator Diagnostics::String"
    name = expr.args[1]
    T = expr.args[2]

    esc(quote
        struct $name <: $Salsa.AbstractAccumulator{$T} end
    end)
end
```

### Step 8: Module integration

**File: `src/Salsa.jl`**

1. Add new exports:

```julia
export @derived, @declare_input, @declare_accumulator, Runtime, DerivedFunctionException
```

2. Include the new file (after `runtime_tracing.jl` but before `default_storage.jl`, since accumulators need `_TracingRuntime` and `_TopLevelRuntime` to be defined, and `DefaultStorage` needs `AbstractAccumulator`):

```julia
include("accumulators.jl")
```

3. Export `push_accumulated!` and `accumulated` as public API functions.

## What Does NOT Need to Change

- **`@derived` macro** — no changes. `push_accumulated!` is a regular function call on the runtime, not a macro.
- **`_TracingRuntime` struct** — stays isbits. Accumulated data lives on the `TraceOfDependencyKeys` (accessed by index).
- **`memoized_lookup`** — no changes to the lookup/unwrap flow itself.
- **`StorageAPI` abstraction** — `accumulated()` can be implemented generically over any storage that stores `DerivedValue`s with the accumulated field. Optionally add `_get_cached_value(storage, key)` to the storage API for custom backends.

## Invalidation Semantics

Accumulated values follow the same invalidation rules as derived values:

| Scenario | Accumulated values |
|----------|-------------------|
| Input unchanged, function not re-executed | Cached accumulated values returned |
| Input changed, function re-executed | Old accumulated values replaced with new ones |
| Input changed, function re-executed but returns same value (early-exit) | Accumulated values are **still replaced** — they come from the new execution |

This last point is important: even if the return value is the same (triggering early-exit for downstream), the diagnostics may differ. Accumulators always reflect the **latest execution**.

## Verification / Test Plan

All tests should be added to a new test file or section in `test/test_salsa.jl`.

1. **Basic push and query**: `@derived` function pushes a diagnostic → `accumulated()` returns it.
2. **Transitive accumulation**: Function A calls function B, B pushes a value, querying `accumulated` on A returns it.
3. **Diamond dependencies**: A calls B and C, both call D which pushes a value — value appears **once** (not twice), thanks to the `visited` set.
4. **Invalidation**: Change an input, re-execute, verify accumulated values update to reflect new execution.
5. **Early-exit interaction**: Function returns the same value after input change but pushes different accumulator values — verify new accumulated values are returned (not stale ones).
6. **Thread safety**: Parallel tasks within a `@derived` function each push accumulated values — all are collected.
7. **Top-level error**: `push_accumulated!(top_level_rt, ...)` throws an error.
8. **Zero-overhead when unused**: Benchmark that functions not using accumulators have negligible overhead (empty dict allocation is cheap; consider benchmarking).
9. **Multiple accumulator types**: Two different accumulator types used in the same function — each queryable independently.
10. **Empty accumulator**: Query an accumulator type that was never pushed to — returns empty vector.

## Design Decisions & Further Considerations

### Ordering
`accumulated()` returns values in execution order (depth-first walk of dependency graph). This matches Rust Salsa's behavior and is the natural result of the recursive `_collect_accumulated!` walk.

### Deduplication
Rust Salsa returns accumulated values as a deduplicated set. This design returns a `Vector` (preserving order and duplicates). This is more flexible — users can deduplicate if needed. If Rust-compatible semantics are desired, change `result` to a `Set` or deduplicate before returning.

### Performance of `accumulated` Dict on `TraceOfDependencyKeys`
For the common case where no accumulators are used, the dict will be empty. An optimization would be to use `nothing` by default and lazily allocate on first `push_accumulated!` call — but a simple empty `Dict` has negligible cost and keeps the code simpler. Consider this optimization if benchmarks show overhead.

### Storage API Extension
For custom storage backends, consider adding `_get_cached_value(storage, key)` to the `AbstractSalsaStorage` trait in `src/storage_api.jl`. This allows `accumulated()` to work with any backend. Alternatively, `accumulated()` can be implemented per-storage-backend if needed.
